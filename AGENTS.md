# Agent memory

## Learned User Preferences

- Git: when pushing changes to the remote repository, push to **`main`** (merge or work on `main` as the default integration branch unless the user explicitly asks for a different branch).
- Prefer slow, smooth UI animations; when the user reports "chunky," "fast without smoothness," or "weird," use spring physics or longer durations with ease-in-out instead of short duration + cubic ease.
- For header or chip collapse flows, include all relevant tabs (e.g. "All") in the animated exit so they fade or collapse together; collapse the group's maxWidth to 0 so hidden elements do not affect layout.
- When an animated element (e.g. plus icon with translateX) escapes its container despite overflow-hidden, prefer removing the transform over adding more overflow or wrapper fixes.
- Collection chips and dashboard segment chips: active state uses `bg-foreground text-background`; inactive state uses `text-foreground` with no background fill â€” not `bg-primary text-primary-foreground`.
- When applying a styling change to one chip variant, apply it consistently to all related variants (All tab, Trash tab, CollectionChip, etc.) without waiting to be asked.
- All clickable UI elements (buttons, interactive chips, clickable wrappers, menu triggers) should explicitly use `cursor-pointer`; disabled actions should use `disabled:cursor-not-allowed` so they clearly look non-interactive.
- The overall UI should lean into a premium, modern, minimal dark-luxury aesthetic (cinematic glass layers, subtle gradients, restrained motion) rather than noisy or heavy visual treatments.
- In settings sections that poll (e.g. Friends), show the loading skeleton only on initial load, not during background refreshes.

## Learned Workspace Facts

- **Env and dev**: Server loads `.env` from `apps/server` via `load-env.ts`. Web client env (e.g. `NEXT_PUBLIC_SERVER_URL`) belongs in `apps/web/.env`; restart the Next dev server after changing it. When running the API from the repo root, Bun `--hot` watches `packages/auth`, `packages/env`, and `packages/db`.
- Better Auth: avoid `authClient.useSession()` in SignInForm and SignUpForm on this stack (Next 16, React 19, Turbopack); use `getSession()` or render the form without session check.
- When the Next.js app and API use different origins (e.g. separate Vercel projects), session cookies are set on the API host; RSC requests to the web origin do not include those cookies. Do not rely on server-side `authClient.getSession({ fetchOptions: { headers: await headers() } })` to see the API sessionâ€”use client-side session checks (e.g. `AuthGate`, `getSession` in the browser) and authenticated `fetch` to the API with `credentials: "include"` (and set `createAuthClient` / server `trustedOrigins` + `CORS_ORIGIN` accordingly).
- TanStack Form: always pass an explicit `selector` prop to `form.Subscribe`; omitting it causes "selector is not a function" at runtime.
- Elysia routes in `apps/server`: Vercel runs TypeScript separately from the bundler (`tsdown`); request `body` is often still typed `unknown` under strict `tsc` even when `t.Object` validates at runtimeâ€”define small interfaces matching each schema and cast `body` in handlers so Vercelâ€™s `tsc` step passes.
- R2 media storage is configured with bucket `kura-media`; credentials (`R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET_NAME`, `R2_PUBLIC_URL`) live in `apps/server/.env`; the S3-compatible SDK endpoint is `https://{R2_ACCOUNT_ID}.r2.cloudflarestorage.com`; `R2_PUBLIC_URL` must be the public-facing `pub-*.r2.dev` (or custom domain) URL, not the API endpoint.
- DB schema lives in `packages/db/src/schema/`; push schema changes with `bun drizzle-kit push` from `packages/db/`.
- Base UI: `ContextMenu.SubmenuTrigger` must be a direct child of `ContextMenu.SubmenuRoot` â€” placing it as a sibling of `ContextMenu.Submenu` throws a runtime error.
- When using `event.currentTarget` in a React setState updater (e.g. checkbox `onChange`), read the value (e.g. `checked`) into a local variable at the start of the handler and use that variable inside the updater; otherwise `currentTarget` may be null when the updater runs.
- Public profile pages use the dedicated domain **`cura.page`**: canonical URLs **`https://<username>.cura.page`**; in-app links use **`getPublicProfileUrl`** from `apps/web/src/lib/public-profile-url.ts`. **`src/proxy.ts`** rewrites `*.cura.page` (and `*.localhost` in dev) to `app/[username]/`, and (unless `NEXT_PUBLIC_PROFILE_PATH_REDIRECT=false` or `VERCEL_ENV=preview`) issues a **308** from `/:username` on the main host to the profile subdomain. Env: **`NEXT_PUBLIC_PROFILE_DOMAIN`** (default `cura.page`).
- Browser extension (`apps/extension`): API base defaults to `http://localhost:3000` in dev and `https://app.kura.so` in production builds; set `WXT_PUBLIC_API_URL` to match `NEXT_PUBLIC_SERVER_URL` when the API is on another host (e.g. `kura-server.vercel.app`). Set `WXT_PUBLIC_WEB_URL` for login/settings links to the Next app. Optional override: `chrome.storage.local` key `kura_api_base`. Popup appearance is stored under `kura_popup_appearance` in `chrome.storage.local` (open via popup gear â†’ `runtime.getURL("/appearance.html")`; `entrypoints/appearance/` avoids `options_ui`). Includes theme, radius (inner controls only â€” Chrome popup shell is square), density, and toggles `showCurrentPage` / `showCollectionPicker` / `showRecent`.
- Polling: followed-collection bookmarks and collection metadata about every 5s; public profile tab content about every 5s and profile header (avatar, banner, counts) about every 10s; Settings â†’ Friends about every 5s for incoming/outgoing requests without reload.
